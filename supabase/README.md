# Database lifecycle (migration-first)

The Supabase Postgres schema is the source of truth for the data layer. It is
managed **migration-first**: every schema change is a versioned SQL file in
`supabase/migrations/`, applied through the Supabase CLI, and TypeScript types
are **generated** from the live schema — never hand-edited.

## One-time setup

```bash
npm install                  # installs the pinned Supabase CLI (devDependency)
npx supabase login           # opens the browser for an access token
npx supabase init            # creates supabase/config.toml (answer N to editor prompts)
npm run db:link              # links to project ref orueodkxqhtbqjddpkrr (asks for the DB password)
```

> Only `202606050001_foundation` was applied by hand through the SQL editor
> before the CLI was linked. `202606050002_user_management` has NOT been applied
> yet. So reconcile **only the foundation** as already-applied, then let the CLI
> push the pending one:
>
> ```bash
> npx supabase migration repair --status applied 202606050001
> npm run db:push      # applies 202606050002_user_management
> ```
>
> After this, `npm run db:push` reports "up to date" and the workflow below is clean.

## Day-to-day workflow

Every schema change follows the same loop:

```bash
npm run db:new add_something   # 1. scaffold a timestamped migration file
#                                2. edit the generated SQL in supabase/migrations/
npm run db:push                # 3. apply pending migrations to the linked project
npm run db:types               # 4. regenerate types/database.ts from the real schema
```

## Scripts

| Script         | Purpose                                                              |
| -------------- | ------------------------------------------------------------------- |
| `db:link`      | Link the repo to the remote Supabase project.                       |
| `db:new`       | Create a new timestamped migration file.                            |
| `db:push`      | Apply pending migrations to the linked project.                     |
| `db:types`     | Regenerate `types/database.ts` from the live schema.                |
| `db:seed`      | Seed development data. Pass the user email: `npm run db:seed -- you@example.com`. |

## Rules

- **Never edit `types/database.ts` by hand.** It is generated by `db:types`.
  If a type looks wrong, fix the schema with a migration and regenerate.
- **Never change the schema from the dashboard** for anything permanent. Author
  a migration so every environment can be rebuilt deterministically.
- Migrations are **append-only**. To change something, add a new migration; do
  not edit a migration that has already been pushed.
- `db:seed` is application-level seeding (it looks up an existing Auth user via
  the service role). It is distinct from Supabase's `seed.sql`, which only runs
  on a local `db reset`.
